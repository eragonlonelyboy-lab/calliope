# Pack: app

Product UI for mobile and web applications: navigation architecture, component systems, states, onboarding, settings, dark mode. The bar here is felt in use, not seen in a screenshot: an app pack verdict is invalid without walking the flows.

## Bar interpretation

1. **Point of view, not a template.** The interaction model takes a position derived from this product's core task. Test: strip the logo and copy; if what remains is tab bar + card list + FAB that could ship as any starter kit, fail. The home screen is the first proof.
2. **Typography that does work.** A declared type ramp with named roles (screen title, section, body, data, caption, control label), anchored to the platform ramp or deliberately diverged with a stated reason. Test: strip all color; every screen still reads in the right order, and data rows scan without borders doing the work.
3. **Restrained color system.** 3-5 colors with named jobs, semantic signals (success/warn/error) defined as system rules, and accent meaning interactive. Test: point at any colored element and name its job; accent on a non-interactive element fails.
4. **Hierarchy that breathes.** One primary action per screen; secondary actions visibly secondary or in overflow. Test: squint test per screen; if two actions tie for winner, fail. Settings screens count: they get hierarchy too, not a flat list of 40 rows.
5. **Imagery with intent.** One icon family, one weight, consistent optical size. Illustration lives only where it teaches (empty states, onboarding) and each instance names what it teaches. Test: caption every illustration with its job; "warms up the screen" fails.
6. **Motion that whispers.** Micro-interactions confirm state change; transitions preserve the spatial model (where did that screen come from, where did it go). Test: every animation names its job (confirm, orient, teach); ambient motion on a screen a user visits 30 times a day fails.
7. **Designed for its surface.** Mobile: primary actions in thumb reach, gestures match platform grammar. Web/desktop: keyboard reachable end to end. Dark mode is a designed mode with its own surface logic, reviewed as its own comp. Test: dark mode comp exists before build, not generated by inverting.
8. **The invisible expensive stuff.** Perceived feedback under 100ms on every tap, every screen has designed empty/loading/error, WCAG AA in both modes, reduced-motion honored, component structure a second developer can extend without archaeology.

## Benchmark

**Linear** (keyboard-first restraint): every action keyboard reachable, one accent doing all interactive signaling, command menu as the navigation spine. Steal: speed as the brand; restraint enforced so hard that any decoration reads as a bug.
**Things 3** (motion as physics): transitions behave like objects with mass; whitespace structures a dense todo domain without visible chrome. Steal: calm under density; motion that explains hierarchy instead of celebrating itself.
**Family wallet** (onboarding): teaches crypto's scariest concepts by doing, one idea per step, progressive disclosure. Steal: first-run and empty states treated as primary product surface, not afterthought.
**Arc browser** (opinionated navigation): breaks browser convention for a coherent spatial model and commits fully. Steal: divergence from platform defaults done with total consistency; half-divergence is the slop version.

## Measurable floors

| Floor | Number | Instrument |
|-------|--------|------------|
| Touch targets | ≥44pt iOS / ≥48dp Android, ≥8pt gap between adjacent targets | overlay measure on smallest supported device |
| Contrast | WCAG AA (4.5:1 body, 3:1 large text and UI components) in BOTH light and dark | contrast checker, both modes |
| Feedback latency | pressed/active state or optimistic update <100ms perceived on every primary action | interaction trace |
| Loading choreography | skeleton (not spinner) when layout is known and load >1s; nothing flashes for loads <300ms | throttled-network walk |
| State coverage | 100% of screens have designed empty, loading, and error (screen x state matrix, no cell blank) | matrix audit |
| Navigation depth | primary tasks reachable in ≤3 taps/clicks from home | depth count per task |
| Reduced motion | prefers-reduced-motion swaps transitions for fades; zero parallax/scale triggers remain | OS toggle walk |
| Frame rate | 60fps on transitions and scrolling (device-profiled, not simulator) | DevTools/Instruments |
| Hygiene | zero console errors (web); zero unhandled error states; offline behavior defined | console + airplane-mode walk |

## QA stations

App packs replace the default web station list in `references/quality/audit_redo_loop.md` with:

1. State matrix walk: every screen x empty/loading/error/offline, no cell blank.
2. Touch target overlay audit against the 44pt/48dp floor.
3. Thumb-zone map on the smallest supported device.
4. Contrast sweep in both light and dark modes.
5. Reduced-motion toggle walk with the OS setting, not a code flag.
6. Latency trace on the three most-used actions.
7. Navigation depth count per primary task.
8. Keyboard walk end to end (web/desktop surfaces).
9. `npx impeccable detect` on web app surfaces.
10. Dark mode side-by-side review against the light comp.

## Technique atlas

| Technique | When to use | Slop version | Implementing skill |
|-----------|-------------|--------------|--------------------|
| Platform anchoring (HIG/Material) | default grammar for nav, gestures, controls; divergence only as a committed system decision (Arc rule) | iOS back-swipe mixed with Material bottom sheets in one app; divergence in one flow only | `taste-skill` |
| Thumb-zone layout | primary actions in bottom third on mobile; destructive actions out of habitual reach | FAB covering the list content it acts on; reachability ignored on tall screens | `taste-skill` |
| Skeleton screens | known layout, load >1s; shapes match the real content geometry | skeleton flashing for 200ms loads; generic gray bars unrelated to final layout | `impeccable` audit |
| Spinners | unknown layout or sub-second indeterminate ops only | full-screen spinner where a skeleton or optimistic update belongs | `impeccable` audit |
| Optimistic UI | high-success-rate ops (rename, toggle, like) with designed rollback | optimistic on payments or destructive ops with no rollback state | manual per floors |
| Navigation depth budget | ≤3 levels; command menu or search as escape hatch for deep content | hamburger hiding primary navigation on mobile; settings as a junk drawer | `ui-ux-pro-max` rubric |
| Dark mode as designed mode | elevation lightens surfaces, accents desaturate, shadows rethought as borders/glow | inverted palette; pure #000 with unadjusted shadows and vibrating saturated accents | `taste-skill` |
| Micro-interaction discipline | confirm state change in ≤200ms; frequency-earned (daily-use elements move least) | spring-bounce on everything; confetti on routine actions | `motion-dev-animations`, `gsap-react` |
| Empty states that teach | first-run empty shows the exact action that fills it, one CTA | sad illustration + "nothing here yet" with no action | `imagegen-frontend-mobile` (concepts) |
| Onboarding by doing | teach inside the product, one concept per step, skippable, progress visible | five-slide swipe tour nobody reads; permission-request wall on first launch | `taste-skill` |

## Treatment flavor

Stylescape = home screen concept + one flow strip (3-4 screens of the core task) + component board (type ramp, color jobs in both modes, core controls in default/pressed/disabled/error) + state board (empty/loading/error for one representative screen) + written motion note (what moves, what job, what stays still). Temperatures: mild = platform-anchored, concept lives in type, color, and density only; medium = one signature interaction + fully designed empty states + custom component skin; hot = custom navigation model and motion as product identity (Things 3 or Arc energy), still inside the floors. Technique palette names atlas entries with the job each does.

## Research notes

Phase 4 adds: platform convention snapshot (what HIG/Material prescribe for this app's pattern, so every divergence is a decision on record); competitor app teardown scored on nav depth and state coverage (shipping designed error states where competitors show raw failures is positioning); device mix and OS version floor for the actual audience (decides the smallest thumb-map device); dark mode adoption share for the audience segment.

## Gate wiring

- **Concept:** `imagegen-frontend-mobile` generates premium screen concepts during treatment (mobile surfaces only; it generates images, never code).
- **Build:** `taste-skill` (layout, type, spacing, anti-slop defaults) → `motion-dev-animations` for React/Next micro-interactions and transitions → `gsap-react` when choreography needs timelines.
- **Audit:** `impeccable` (`npx impeccable detect` + audit command, 27 anti-patterns) → `ui-ux-pro-max` rubric-mode ONLY after the design system is locked; NEVER `--design-system` on a locked system.
- **Polish:** `impeccable` one-word commands (quieter, bolder, polish) driven by CD verdict language.
- Missing skill → the station's check runs manually against this pack's floors; note the degradation in the QA report.

## Pre-flight (builder self-check)

1. State matrix filled: every screen has empty/loading/error designed, none improvised at runtime.
2. Touch targets measured on the smallest supported device.
3. Thumb map drawn; primary actions in reach, destructive actions out of it.
4. Every tap shows feedback under 100ms.
5. Dark mode reviewed as its own comp; both modes pass AA.
6. Reduced-motion path walked with the OS toggle.
7. Primary tasks counted at ≤3 taps from home.
8. Type ramp declared; zero orphan sizes in computed styles.
9. Every color names its job; accent only on interactive elements.
10. Icons one family, one weight.
11. Every empty state contains the action that fills it.
12. Onboarding skippable and teaches by doing.
13. Offline and airplane-mode behavior defined, not accidental.
14. Keyboard walk completes (web/desktop).
15. Destructive actions confirmed or undoable; never both absent.
16. Zero console errors; component structure navigable by a stranger.
